# Benchmark data

`corpus.csv` is the checked-in inventory of selected, unique PyPTO and
PyPTO-Lib DSA instances. It includes both solver-search workloads and
compiler/constraint coverage cases. It is generated by `dsa-suite` as
`features.csv`; one row describes one structured JSON input.

The main columns cover provenance, buffer count, minimum/maximum/total buffer
bytes, number of distinct sizes, whether every buffer has the same size,
alignment, temporal conflict and reuse-candidate counts,
memory-space capacities, raw interval-live bytes, and PyPTO alias/pipeline
structure. `uniform_buffer_size=true` is the unweighted/unit-weighted special
case of DSA.

The corpus excludes captures with no placement choice: instances whose buffers
can all share one address and that have no explicit separation, pinning,
colocation, temporal-exclusion, or reuse-cost structure. Zero temporal
conflicts alone is not an exclusion criterion because PyPTO-specific
constraints can still make such an instance meaningful.

PyPTO pool names map to hardware memory spaces as follows:

| Pool | Hardware space |
| --- | --- |
| `Vec` | UB |
| `Mat` | L1 |
| `Left` | L0A |
| `Right` | L0B |
| `Acc` | L0C |

A document may contain several independent pools. Therefore
`pool_capacities_bytes` and `max_interval_live_bytes_by_space` retain one value
per space instead of collapsing L1 and the three L0 memories into one number.
`max_live_capacity_ratio` is the largest raw interval-live/capacity ratio among
the document's pools; it is a pressure indicator, not a solved placement peak.

`capture/` contains coverage specifications used to create the corpus, while
`pypto/` and `pypto-lib/` contain the actual solver inputs.

## Why a matmul document has four pools

A cube matmul normally touches four physically separate address spaces:
`Mat`/L1, `Left`/L0A, `Right`/L0B, and `Acc`/L0C. For example,
`pypto::tests::st::runtime::ops::test_matmul::pypto_matmul` contains:

| Pool | Buffers | 910B usable capacity | Raw live-byte peak |
| --- | ---: | ---: | ---: |
| L1 | 2 | 524,288 B | 65,536 B |
| L0A | 1 | 65,536 B | 32,768 B |
| L0B | 1 | 65,536 B | 32,768 B |
| L0C | 1 | 131,072 B | 65,536 B |

These are one compiler instance because they come from one scheduled kernel,
but they are independent address-allocation problems: offsets in L1 cannot
overlap or consume capacity in L0A, L0B, or L0C. With fixed pool membership,
the feasible set is the product of the four per-pool feasible sets, so the
document can be solved pool by pool. The multi-pool envelope is beyond the
single-pool `standard_dsa` file format, but does not itself introduce new
combinatorial coupling. Flexible pool assignment, a shared capacity budget, or
joint pipeline-depth decisions would introduce genuine cross-pool coupling.

The plain five-buffer matmul above is primarily a compiler-coverage case, not a
difficult search instance: three pools contain one buffer and the two L1
buffers conflict. Matmul-derived cases with reuse candidates, pipeline groups,
or explicit separations are the meaningful algorithmic cases. Per-pool
`pypto_core_relaxation` documents additionally remove PyPTO constraints and are
lower bounds; partitioning fixed pools alone is exact and should not be
confused with that relaxation.

## Program and architecture

The current JSON is already a *bound* problem: program-derived buffers and
lifetimes are stored together with target capacities. The intended general
model is instead:

```text
benchmark instance = lowered program variant + architecture specification
bound DSA problem   = Bind(lowered program variant, architecture specification)
```

An architecture specification supplies usable capacities, minimum alignments,
reserved ranges, and later bank geometry for the logical spaces UB, L1, L0A,
L0B, and L0C. A program variant supplies buffer sizes, logical spaces,
lifetimes, and constraints. Reports should key results by both fingerprints,
so a 910B and 950 binding of the same program are paired observations rather
than unrelated instances.

This does **not** mean every 910B capture may be relabeled as 950 by changing
capacities. PyPTO performs target-dependent lowering before DSA export; for
example, 950 has a larger L0C and can choose a different matmul tiling, while
cross-core and write-back lowering also differ. Capacity-only rebinding is an
actual target case only when the lowered program fingerprint is unchanged.
Otherwise each target needs its own lowered program variant. See
[`architecture_binding.md`](../docs/architecture_binding.md) for the proposed
separation and binding contract.
